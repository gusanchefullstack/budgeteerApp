# Use cases in UI
Considering the @backend API developed for Budgeteer APP, the following use cases must be implemented in the UI:

## Registering new user
The UI must provide password confirmation (input twice) for being sure the password is the intended by user

## Login User
The UI must allow to login with user name and password. Password musn't be shown clear text.
Link for registering if not registered yet
Provide validation of input fiels
Provide correct message of invalid username/password.
If sucessful login, user must be redirected to page when show her current budget (if created) or start the creation of new one
Token must be received and stored

## Logout User
Token must be deleted and redirected to login page

## Delete Account 
Account/User deleted in server including its budget and transactions

## Create Budget
Three alternatives for budget Creation

1. User create a once complete tree (visually appealing) for creating budget, basic data (name, beginning and end date), categories in expenses and incomes, budget groups, budget items. All info is sent to the API in "one shot"
2. User create a budget step by step guided by UI. Here you could use tools described in stack. First basic data, then incomes -> category ->budget groups -> budget items for each category, then expenses -> category ->budget groups -> budget items for each expense category and so on.
3. User create budget only with basic data and then he will be able to add categories/budget group/items in another moment.

## Get Budget
Show budget in a visual format with all of its elements (incomes, expenses, categories, groups, items)
The page where budget is shown must include the option to fetch and show associated transactions based on filters and must be paginated.

# Update Budget
Update budget data and name of categories, groups, items. This operation must change the corresponding fields in associated transactions in backend api

# Delete Budget
Delete budget and tree including transactions. Note that transactions cannot be deleted by user directly.

# CRUD for Budget Categories
It can be done separately once the budget is created. In Update operation backend app must change the corresponding fields in associated transactions. In delete operation, associated hierarchy and transactions must be deleted

# CRUD for Budget Groups
It can be done separately once the budget and parent category are created.In Update operation backend app must change the corresponding fields in associated transactions. In delete operation, associated  hierarchy and transactions must be deleted. Parent elements (category) must be only valid (already created)

# CRUD for Budget Items
It can be done separately once the budget, parent category and group are created. In Update operation backend app must change the corresponding fields in associated transactions. In delete operation, associated hierarchy and transactions must be deleted. The frequency must restrict to options available in API. The planned dates must be validated for belonging to begining and end date of Budget period. Parent elements (category/Group) must be only valid (already created)
